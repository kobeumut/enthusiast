# Retrieval quality phase (YAZ-15)

This document describes the optional RAG-quality retrieval stages added to the product and document
retrievers, the config knobs that enable them, and the benchmark-driven verification that ships with
them. The goal is to make every retrieval-quality lever **explicit, config-driven, and off by
default**, so the historical pure-vector behaviour is preserved exactly until a dataset opts in.

## Background

Before this phase, both retrievers used a single path: embed the query, rank chunks by cosine
distance, return the top-K. That baseline is correct but leaves quality on the table for queries
that carry explicit terms (SKUs, model numbers), implicit scope (categories, price, source), or
duplicate-prone content (documents split into overlapping chunks). The DRA's priority order for this
phase was: metadata filtering → hybrid standardisation → reranking → MMR/diversity → HNSW tuning.

## The five stages (all opt-in)

Every stage is disabled by default. They compose in a fixed, sensible order inside the retriever:

```
query
  → embed → vector ranklist (HNSW, ef_search-tunable)
  → [hybrid] fuse with full-text keyword ranklist via Reciprocal Rank Fusion
  → [rerank] lexical rerank over the candidate pool
  → [MMR]    diversity selection (documents only) over the candidate embeddings
  → distinct products / top-K chunks
```

Stages act on a **candidate pool** (default 50–60 chunks) rather than the final top-K, so the
reranker/diversity step has room to promote a good candidate the vector ranker placed just outside
the final cut.

### 1. Metadata filtering (`filters=`)

Pushes category/price (products) or url/title (documents) predicates into the chunk queryset
*before* the vector ranking, so out-of-scope chunks never enter the ranking. Implemented in
`agent/core/retrievers/filters.py` as a `RetrievalFilters` value object translated to Django `Q`
predicates. Passed per-call to the retriever:

```python
from agent.core.retrievers.filters import RetrievalFilters

retriever.find_products_matching_query(
    "running shoes",
    filters=RetrievalFilters(categories=["Running"], price_max=100),
)
```

`filters=None` (the default) applies no predicate.

### 2. Hybrid retrieval (`hybrid_enabled`)

Fuses the vector ranklist with a PostgreSQL full-text keyword ranklist using **Reciprocal Rank
Fusion** (`agent/core/retrievers/hybrid.py`, RRF `k=60`). RRF needs only ranks, not calibrated
scores, so it combines cosine-distance and `ts_rank` lists without weight tuning. This recovers
exact-term hits (model numbers, error codes) the embedding space blurs.

The keyword ranklist is generated by the new `get_chunks_by_keyword_for_data_set` repository method,
which filters on `SearchVector("content") @@ SearchQuery(keyword)` (the `@@` match, not
`ts_rank > 0`, to avoid `ts_rank`'s near-zero noise leaking non-matches into the ranklist).

### 3. Reranking (`reranker_enabled`)

Applies a cheap second-stage reranker over the candidate pool — the single highest-leverage RAG
quality improvement. The default `LexicalReranker` (`agent/core/retrievers/reranking.py`) blends
vector similarity (from the annotated `.distance`) with lexical coverage (fraction of query tokens
present in the chunk content). A cross-encoder / LLM reranker can be added later by subclassing
`BaseReranker` and injecting it via the `reranker=` kwarg; the retriever wiring is agnostic to the
implementation. Reranking never drops candidates — it only reorders.

### 4. MMR diversity (`mmr_enabled`, documents only)

Selects relevant-yet-diverse chunks via **Maximal Marginal Relevance**
(`agent/core/retrievers/diversity.py`) so one document's near-duplicate sections do not crowd out
other sections the user needs. `mmr_lambda` (default 0.5) trades relevance against novelty: `1.0` =
pure relevance (no diversity), `0.0` = pure novelty. Implemented in pure Python over the candidate
embeddings (already fetched for rerank), so it adds no extra DB round-trips.

### 5. HNSW tuning (`ef_search`)

Runs `SET LOCAL hnsw.ef_search = N` inside a transaction before the vector query
(`_materialise_with_hnsw_ef_search` in `agent/core/repositories.py`), tuning the HNSW candidate-list
size at runtime without rebuilding the index. `ef_search` is a runtime GUC (default 40); larger
values trade latency for recall.

## Enabling stages per dataset

Stages are wired through `RetrieverConfig.extra_kwargs` in `agent/core/agents/default_config.py`,
so they are configured alongside the existing `number_of_products` / `max_objects` knobs. To opt a
dataset/agent into the full pipeline:

```python
retrievers=RetrieversConfig(
    product=RetrieverConfig(
        retriever_class=ProductRetriever,
        extra_kwargs={
            "number_of_products": 12,
            "hybrid_enabled": True,
            "reranker_enabled": True,
            "ef_search": 100,
        },
    ),
    document=RetrieverConfig(
        retriever_class=DocumentRetriever,
        extra_kwargs={
            "max_objects": 12,
            "hybrid_enabled": True,
            "reranker_enabled": True,
            "mmr_enabled": True,
            "mmr_lambda": 0.5,
            "ef_search": 100,
        },
    ),
)
```

## Verification (benchmark-driven)

Three test layers ship with this phase. All run against a real pgvector PostgreSQL instance (the CI
service) — no mocks of the vector store.

| Layer | File | What it proves |
|---|---|---|
| Pure-module units | `agent/core/retrievers/tests/test_{hybrid_rrf,filters,reranking,diversity}.py` | RRF formula, filter translation, rerank blend, MMR selection — fast, no DB |
| Integration | `tests/test_{metadata_filters,retriever_quality_stages}.py` | Each stage moves the result ordering in the intended direction against real pgvector data; defaults preserve pure-vector order |
| EXPLAIN | `catalog/tests/test_pgvector_explain.py` | HNSW index is used for the pure vector NN query; runtime `ef_search` is accepted; the dataset-filter interaction is characterised |
| Eval harness | `agent/core/retrievers/eval/test_eval_retrieval_quality.py` | Before/after Precision@K / Recall@K / MRR on a labelled corpus (prints a summary table) |

### Eval harness output (baseline → stage enabled)

Each stage improves its targeted metric on the labelled corpus.

## Important finding: the dataset-filter ↔ HNSW interaction

EXPLAIN captured that the **production retrieval query does not use the HNSW index**: it joins the
chunk table to its parent (`document`/`product`) and filters by `data_set_id`, a relational
predicate the embedding HNSW index cannot serve. The planner therefore scans chunks reachable via
the foreign-key index and sorts them by cosine distance — even with `enable_seqscan = off`. (The
*pure* nearest-neighbour query, with no dataset join, does use the HNSW index, which the EXPLAIN
test pins.)

**Consequence:** on the current schema, `ef_search` only takes effect once the HNSW index is in the
plan, so it has no effect on the production dataset-scoped path today. The stages that deliver the
real lift on the current schema are **metadata filtering, hybrid RRF, reranking, and MMR**.

The robust fix — denormalising `data_set_id` onto the chunk tables and adding a per-dataset partial
HNSW index, or a two-stage retrieve-then-filter — is called out as the natural follow-up; the
EXPLAIN test that asserts the negative will flip to a positive assertion the day that change lands.
